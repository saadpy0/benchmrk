Phase 1: Core Database & Verification Schema
We need to extend the current Prisma schema to support continuous monitoring and held balances.

CreatorBaseline table: Stores the historical average of a creator (views, engagement ratio) updated weekly.
MetricSnapshot table: A time-series table attached to submissions. Every time we poll the API, we insert a row (timestamp, view_count, likes, comments).
Ledger or Balance table: Track pending_balance (funds in the 10-day hold) and available_balance (funds cleared for withdrawal).
Phase 2: Gate 1 — The Baseline Profiler
Goal: Establish what "normal" looks like for the creator.

When a creator joins/connects their account, we pull their last 30 posts.
Calculate and store their historical_engagement_ratio and average_views.
Assign an initial trust_score based on account age and baseline consistency.
Phase 3: The Continuous Tracking Engine (The Core)
Goal: Automate the polling of views and engagement over time.

Trigger: Creator submits the video link right after posting.
Workflow (via Trigger.dev or cron job):
The system automatically polls the API at set intervals (e.g., Hours 6, 12, 24, 72, and daily thereafter).
Every pull logs a MetricSnapshot.
The system continuously calculates the current engagement ratio and compares the view growth curve against typical patterns.
Phase 4: Gate 2 & 3 — Authentication & The 10-Day Hold
Goal: Verify views are real and move funds from pending to available.

As views accumulate, the theoretical payout is calculated and added to the creator's pending_balance.
The 72-Hour Check: At hour 72, the system runs the heavy authentication logic (Gate 2):
Velocity Check: Did views spike unnaturally?
Engagement Check: Does the ratio match the baseline?
Drop Check: Did the platform scrub views?
The Score: The submission is scored (Gate 3). If it passes, it is marked as "Verified."
The 10-Day Maturation: Funds generated by verified views sit in pending_balance for exactly 10 days from the moment those views were recorded. On day 10, a final check confirms views didn't drop significantly, and the funds are moved to available_balance for withdrawal.
Phase 5: Dashboards & Review Queue
Goal: Provide human oversight for edge cases.

Admin Review Queue: Submissions that score between 40-70 in the 72-hour check are paused. The admin dashboard surfaces the MetricSnapshot graphs (velocity curves, engagement ratios) so a human can easily spot bot patterns and manually approve or reject.
Creator Dashboard: Show creators their real-time accumulating views, pending_balance (with countdowns until clearance), and available_balance.
This roadmap builds a continuous, automated pipeline where the creator does almost nothing after submitting, and the system handles the fraud-checking and financial holds automatically.
